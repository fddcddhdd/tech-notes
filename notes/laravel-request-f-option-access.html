LaravelのRequestで$request->f_optionと$request['f_option']は何が違うのか

作成日: 2026-06-08

ひとことで言うと

Laravel 13.x の実装では、値を読むだけなら $request->f_option と $request['f_option'] はかなり近い動きをします。 ただし、初心者には $request->input('f_option') のような明示的なメソッドを使う方が分かりやすく、安全です。

背景

Laravelのコントローラでは、フォームやクエリ文字列、JSONリクエストなどから送られた値を Illuminate\Http\Request オブジェクト経由で取り出します。 たとえばフォームに name="f_option" という入力項目がある場合、次のような書き方を見かけることがあります。 

<?php

public function store(Request $request)
{
    $value1 = $request->f_option;
    $value2 = $request['f_option'];
    $value3 = $request->input('f_option');
}

見た目は違いますが、どれも「リクエストから f_option という名前の値を取り出したい」という意図で使われます。 ただし、それぞれの仕組みと読みやすさは同じではありません。

詳しい説明

$request->f_option は「動的プロパティ」

$request->f_option は、Requestオブジェクトの本当のプロパティを読んでいるように見えます。 しかしLaravelでは、存在しないプロパティを読もうとしたときに、リクエスト入力から同名の値を探す仕組みが用意されています。

Laravel 13.x の公式ドキュメントでは、動的プロパティで入力値を取得でき、入力値が見つからない場合は一致したルートパラメータを探すと説明されています。 つまり $request->f_option は、単なるオブジェクトのプロパティ参照ではなく、Laravelが裏側で入力値を探してくれる省略記法です。

$request['f_option'] は「配列アクセス」

$request['f_option'] は、$request を配列のように扱っている書き方です。 ただし $request 自体が普通の配列になるわけではありません。 LaravelのRequestはPHPの ArrayAccess という仕組みに対応しているため、配列のような記法で値を読めます。

Laravel 13.x のソースでは、配列アクセスで値を読む処理 offsetGet() が、動的プロパティ取得の処理 __get() を呼ぶ形になっています。 そのため、少なくとも値を読む場面では $request['f_option'] も $request->f_option とほぼ同じ経路で値を取りに行きます。

明示的に書くなら input() が分かりやすい

実務では、次のように input() を使うと意図が伝わりやすくなります。 

<?php

$value = $request->input('f_option');

input() なら、第2引数でデフォルト値も指定できます。 

<?php

$value = $request->input('f_option', '未指定');

また、真偽値として扱いたい場合は boolean()、配列として扱いたい場合は array() のように、 目的に合ったメソッドを選ぶと、読み手にも型の意図が伝わります。

具体例

読むだけなら近い結果になる例

たとえば次のようなフォームから f_option=on が送られてきたとします。 

<input type="checkbox" name="f_option" value="on">

Laravel 13.x では、次の3つはどれも f_option の値を取り出す目的で使えます。 

<?php

$a = $request->f_option;
$b = $request['f_option'];
$c = $request->input('f_option');

ただし、あとから読む人にとって一番分かりやすいのは $request->input('f_option') です。 「リクエスト入力を読んでいる」と明確に伝わるためです。

デフォルト値が必要な例

チェックボックスは、チェックされていない場合に値自体が送られないことがあります。 そのため、未送信の場合の扱いを決めておくと安全です。 

<?php

$option = $request->input('f_option', 'off');
$enabled = $request->boolean('f_option');

このように書くと、「値がなかった場合はどうするか」「真偽値として読むのか」がコード上で分かります。

ルートパラメータと同じ名前の場合

公式ドキュメントによると、動的プロパティは入力値が見つからない場合、マッチしたルートパラメータも探します。 たとえば次のようなルートがあるとします。 

<?php

Route::get('/settings/{f_option}', function (Request $request) {
    $fromProperty = $request->f_option;
    $fromArray = $request['f_option'];
    $fromInput = $request->input('f_option');
});

リクエスト入力に f_option がない場合、$request->f_option と $request['f_option'] はルートパラメータの値を返す可能性があります。 一方で、$request->input('f_option') は入力値を読むためのメソッドなので、ルートパラメータを読みたい場合は $request->route('f_option') と分けて書く方が明確です。

よくある勘違い

勘違い1: $request['f_option'] と書けるからRequestは配列である

Requestは普通の配列ではありません。 PHPの ArrayAccess に対応しているため、配列のような記法を使えるだけです。 配列として扱い切れると考えると、存在確認や書き込みの挙動で混乱しやすくなります。

勘違い2: どの書き方でも完全に同じである

値を読むだけなら近い結果になりやすいですが、完全に同じ意味だと考えない方が安全です。 特に、デフォルト値を指定したいとき、真偽値として読みたいとき、クエリ文字列だけを読みたいとき、ルートパラメータだけを読みたいときは、 input()、boolean()、query()、route() などの専用メソッドを使う方が明確です。

勘違い3: チェックボックスが未チェックなら必ず false が入る

HTMLフォームでは、未チェックのチェックボックスはそもそも送信されないことがあります。 そのため f_option が存在しない場合があります。 Laravel側では $request->boolean('f_option') や $request->input('f_option', 'off') のように、 未送信のケースを意識して書くと読みやすくなります。

勘違い4: $request->f_option は常にフォーム項目だけを見る

Laravel 13.x の公式ドキュメントでは、動的プロパティは入力値がなければルートパラメータも探すと説明されています。 フォーム項目だけ、クエリ文字列だけ、ルートパラメータだけを明確に読みたい場合は、それぞれ専用メソッドで分けて書くのが安全です。

まとめ

$request->f_option は動的プロパティ、$request['f_option'] は配列アクセスです。 Laravel 13.x では、読み取り時にかなり近い経路で値を取得します。 ただし、初心者が読み返すコードや、将来の保守を考えるコードでは、$request->input('f_option') や $request->boolean('f_option') のように、意図がはっきりするメソッドを使うのがおすすめです。
